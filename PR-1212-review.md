## Executive Summary

This is a **major architectural refactor** of DynamicPPL.jl that fundamentally changes how variable transformations (linking) are managed. The PR moves from in-place VarInfo modifications to an accumulator-based system, introducing:

- **AbstractTransformedValue hierarchy** (VectorValue, LinkedVectorValue, UntransformedValue)
- **AbstractLinkStrategy pattern** (LinkAll, UnlinkAll, LinkSome, UnlinkSome)
- **VarNamedTuple-backed VarInfo** replacing both SimpleVarInfo and VarNamedVector
- **OnlyAccsVarInfo** for lightweight accumulator-only evaluation

**Stats**: +7,527/-7,610 lines across 85 files, targeting the `breaking` branch

## Issues by Severity

### CRITICAL (3 Issues)

**1. Complex LogJac Adjustment Logic is Fragile**
- **Location**: `src/accs/transformed_values.jl:120-127`
- **Problem**: The adjustment `linker.logjac += vect_logjac - logjac` assumes precise semantics of the incoming `logjac` parameter. The comment states "in principle should be zero, but..." indicating uncertainty.
- **Risk**: Incorrect log-Jacobian corrections would cause silently wrong posterior samples.
- **Recommendation**: Add explicit unit tests verifying: `link(invlink(vi)) == vi` and that accumulated logjac matches expected transformation Jacobian.

**2. Dual LogJac Accumulation Creates Confusion**
- **Location**: LogJacobian is accumulated in BOTH `LogJacobianAccumulator` AND `Link!.logjac`, then combined at `varinfo.jl:135`
- **Problem**: Understanding how these interact requires reading multiple files and following complex control flow.
- **Risk**: Future maintenance could introduce double-counting bugs.
- **Recommendation**: Add clear documentation or refactor to single-path accumulation.

**3. Stale Cached Transform is Technical Debt**
- **Location**: `src/transformed_values.jl:84-90`
- **Problem**: The `transform` field in VectorValue/LinkedVectorValue is explicitly documented as potentially inaccurate after `unflatten!!`. The codebase works around this by ALWAYS recomputing transforms from distributions.
- **Risk**: Dead code complexity; if anyone trusts the cached transform, they'll get wrong results.
- **Recommendation**: Either remove the field entirely, or implement a validity tracking mechanism.

---

### IMPORTANT (5 Issues)

**4. Missing Migration Guide for Breaking Changes**
- **Problem**: Major types removed (SimpleVarInfo, VarNamedVector, Experimental module), function signatures changed, but no migration guide.
- **Impact**: Downstream packages (Turing.jl, AdvancedHMC.jl) will face undocumented breakages.
- **Recommendation**: Add explicit migration guide with old->new API mapping.

**5. Template Argument Propagation Underdocumented**
- **Location**: `src/compiler.jl:541-546`
- **Problem**: The new `template` argument to `tilde_assume!!` and `accumulate_assume!!` is auto-generated by the compiler. External packages implementing custom contexts won't know to update.
- **Recommendation**: Document this explicitly in HISTORY.md and api.md.

**6. VNTAccumulator Creates New Objects on Every Assume**
- **Location**: `src/accs/vnt.jl:49-55`
- **Problem**: Each `accumulate_assume!!` creates a new VNTAccumulator and VarNamedTuple. For models with N variables, this is O(N) small allocations.
- **Mitigation**: Type-stable, small allocations suitable for optimization.
- **Recommendation**: Benchmark to verify this doesn't cause performance regression on large models.

**7. Test Coverage Gaps for LinkSome/UnlinkSome**
- **Location**: `test/varinfo.jl:348-366`
- **Problem**: LinkSome/UnlinkSome are tested for correct linking status, but NOT for correct inference behavior afterward.
- **Recommendation**: Add integration tests that sample from models using partial linking strategies.

**8. VNTAccumulator Not Explicitly Tested**
- **Location**: Missing from `test/accumulators.jl`
- **Problem**: New accumulator type with critical role in linking is not directly tested.
- **Recommendation**: Add dedicated test suite for VNTAccumulator behavior.

---

### MINOR (6 Issues)

**9. AbstractDict Indexing Removal Needs Documentation**
- HISTORY.md line 13-26 documents this, but it's a subtle breaking change that may surprise users.

**10. `get_transform` and `get_internal_value` Exported Without Docstrings**
- **Location**: `src/DynamicPPL.jl:126-127`
- Functions are public but lack inline docstrings.

**11. OnlyAccsVarInfo Semantic Awkwardness**
- A "VarInfo" that doesn't store values is conceptually odd. Consider renaming to `AccumulatorPhaseState`.

**12. Three-Way Dispatch in Link! is Complex**
- Three overloads of `Link!` callable with different logjac handling for each case. Consider unified approach.

**13. Link Strategies Lack Practical Usage Examples in Docstrings**

**14. API Documentation Structure**
- New concepts scattered across multiple sections without narrative cohesion.

## Strengths

### Architecture
1. **Avoids In-Place VarInfo Modification**: Cleaner, more functional approach to linking
2. **Flexible Link Strategy Pattern**: Clean API (`VarInfo(model, LinkAll())`)
3. **Immediate Linked VarInfo Creation**: Can now create linked VarInfo directly without first creating unlinked
4. **OnlyAccsVarInfo Minimalism**: Reduces per-evaluation allocations during likelihood computation

### Performance
1. **Unified VarNamedTuple**: Eliminates NamedTuple + Dict dual representation overhead
2. **Type-Stable Design**: VarNamedTuple enables aggressive AD specialization
3. **Simplified LogDensityFunction**: Fewer struct fields and constructor arguments

### Documentation
1. **Excellent flow.md**: Clear three-stage model explanation (Initialisation -> Computation -> Accumulation)
2. **Comprehensive VNT Docs**: 4-file deep dive into VarNamedTuple design and motivation
3. **Good Docstrings**: AbstractTransformedValue subtypes well-documented with warnings

### Testing
1. **Comprehensive VNT Tests**: 1626 lines in varnamedtuple.jl with good edge case coverage
2. **Type Stability Focus**: Extensive `@inferred` usage throughout test suite
3. **Helper Functions**: Well-designed test utilities (`test_invariants`, `test_link_strategy`)

## Verdict

### APPROVE WITH RESERVATIONS

This is a well-designed architectural improvement that addresses a real problem (the complexity of VarInfo linking). The accumulator-based approach is cleaner and more maintainable than in-place modifications.

**However, the following should be addressed before merge:**

1. **Critical**: Add property-based tests verifying logjac correctness (`link . invlink == id`)
2. **Important**: Add migration guide to HISTORY.md
3. **Important**: Add VNTAccumulator test coverage
4. **Important**: Document template argument in function signatures

**Post-merge recommendations:**
- Consider removing stale transform caching (simplify AbstractTransformedValue)
- Add performance benchmarks comparing old vs new linking performance
- Consider renaming OnlyAccsVarInfo for semantic clarity

The PR represents solid work addressing issue #836 and should significantly improve the maintainability and performance of DynamicPPL's transformation system once the critical issues are addressed.
